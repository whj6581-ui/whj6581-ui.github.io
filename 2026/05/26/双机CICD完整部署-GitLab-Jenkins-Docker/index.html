从零搭建 CI/CD 流水线：GitLab + Jenkins + Docker 实战记录

本文记录了我独立搭建一套双机 CI/CD 自动化部署平台的完整过程，涵盖环境规划、工具配置、流水线编写、踩坑排查，适合有一定 Linux 和 Docker 基础的同学参考。

背景与目标

在学习 DevOps 的过程中，我希望搭建一套尽可能贴近企业真实环境的 CI/CD 平台，而不只是在单机上跑个 demo。最终目标是：

代码推送 → 自动触发构建 → 自动部署上线 → 自动健康验证，全程零人工干预。

为此我规划了双机分离的架构：一台跑 GitLab 负责代码托管，另一台跑 Jenkins 负责构建和部署，两台机器通过 SSH 和 Webhook 打通。

两台机器均为 CentOS 7.9，运行在 VMware 虚拟机中，网络使用 NAT 模式保证互通。

所有服务均以 Docker 容器方式部署，便于迁移和复现。

技术选型

  • GitLab CE：自托管的代码仓库，支持 Webhook，无需依赖 GitHub/Gitee
  • Jenkins：成熟的 CI/CD 引擎，Pipeline as Code，通过 Jenkinsfile 声明式定义流水线
  • Docker + Docker Compose：容器化应用，Compose 负责多服务编排
  • Flask + MySQL + Nginx：作为被部署的应用栈，Nginx 反向代理 Flask，Flask 连接 MySQL

踩坑记录

1. 容器内 docker 命令找不到

现象： Jenkins 流水线报 docker: not found

原因： Jenkins 官方镜像不包含 Docker，挂载了 docker.sock 但没有挂载二进制文件

解决： 启动 Jenkins 容器时额外挂载 /usr/bin/docker 和 cli-plugins 目录

2. docker compose 子命令不可用

现象： docker compose version 报错，子命令不识别

原因： cli-plugins 路径在不同系统不一样，我的机器实际路径是 /usr/libexec/docker/cli-plugins，而不是 /usr/lib/docker/cli-plugins

解决： which docker-compose 和 ls /usr/libexec/docker/cli-plugins 确认路径后，挂载正确路径

3. 健康检查 curl localhost 失败

现象： 流水线最后一步 curl http://localhost/health 始终返回连接拒绝

原因： 健康检查命令在 Jenkins 容器内执行，容器内的 localhost 指向容器自身，而 Nginx 运行在另一个容器里，端口映射在宿主机上

解决： 改用宿主机 IP curl http://192.168.2.150/health，从宿主机的端口映射访问 Nginx

总结

这套环境从零搭建下来，踩了不少坑，但每一个问题的排查过程都加深了我对 Docker 网络模型、容器权限、SSH 认证机制的理解。

几个印象最深的点：

  • 容器的网络隔离：容器内的 localhost 和宿主机的 localhost 完全不同，这个认知很基础但容易忽略
  • Servlet 规范的版本演进：javax → jakarta 的迁移是 Java 生态的一个重要断层，版本选型要格外注意
  • JSON 格式的严格性：配置文件的一个多余逗号就能让整个服务起不来，细节很重要

后续计划把这套环境迁移到 Kubernetes，用 K8s 替代 Docker Compose 做容器编排，进一步完善 DevOps 技能栈。
