四、为什么哨兵节点数量至少要3个

哨兵集群通过”多数派”机制判断 Master 是否真的宕机，避免误判（脑裂问题）：

  • 1个哨兵：单点故障，哨兵自己挂了就没人监控
  • 2个哨兵：投票可能打平（1 vs 1），无法达成共识
  • 3个哨兵：任意1个故障，剩下2个仍能达成多数派共识（2/3）

这也是生产环境哨兵节点数量通常是奇数（3、5、7…）的原因。

五、故障转移实测

模拟 Master 宕机

查看哨兵日志，完整记录了故障转移全过程

日志解读：

日志 含义
+sdown 主观下线，单个哨兵认为Master无响应
+odown 客观下线，达到quorum=2的要求，多数哨兵确认
+switch-master 故障转移完成，新Master确定为111

整个过程在几秒内自动完成，没有人工介入。

六、踩坑现象记录：节点重启后角色丢失之谜

这是整个实验中最有价值的一段排查过程。

现象

故障转移完成后（Master已切换为111），把三台机器的 Redis 和哨兵进程全部关闭重启，预期所有节点应该自动恢复到故障转移后的正确角色，但查看 107 的状态：

107 变成了一个孤立的 Master，既不是从节点，也没有从节点跟着它，集群关系看起来完全丢失了。

排查过程

第一步怀疑是哨兵的运行时变更没有持久化到配置文件，重启后所有节点都按配置文件里的旧角色启动。但检查 110 的配置文件发现：

配置文件里的 replicaof 已经被自动改写成了新Master（111），不再是最初写的107！进一步验证 110 和 111 的实际运行状态：

说明 110 和 111 之间的主从关系完全正确，且重启后依然保持，问题只出在 107 身上。

根本原因

哨兵在执行故障转移时，会通过 CONFIG REWRITE 机制，自动把所有存活节点的配置文件中的 replicaof 字段更新为新Master地址，这正是110能在重启后自动正确连接111的原因。

但 107 在故障转移发生时已经宕机了，它没有”活着”去接收哨兵的 CONFIG REWRITE 指令。哨兵只能修改它能连接到的、还在运行的节点的本地配置文件，无法修改一个离线节点的磁盘文件。所以107的配置文件永远停留在故障前的状态——它最初就是Master，配置文件里从未写过 replicaof，重启后自然以独立Master身份启动。

这不是哨兵的缺陷，而是完全符合预期的行为边界：哨兵管理的是”活着的”集群拓扑，离线节点重新上线后不会自动恢复角色，需要哨兵重新发现并接管，期间需要手动或等待介入。

七、这次排查得到的关键结论

  1. 哨兵的 CONFIG REWRITE 机制只能同步”活着”的节点，离线节点的配置文件不会被自动更新，重新上线后需要重新被哨兵发现和接管。
  2. 手动用 redis-cli 执行的 replicaof 命令只是运行时变更，但哨兵在故障转移时执行的 replicaof 切换会进一步触发 CONFIG REWRITE 持久化到配置文件，这是哨兵和手动操作的本质区别。
  3. 节点重新上线加入集群时，要确认 masterauth 配置是否齐全，否则即使角色正确切换，也会因为认证失败导致 master_link_status:down，数据无法真正同步。
  4. 排查主从问题的标准流程：先看 info replication 确认 role 和 master_host，再看 master_link_status 确认连接是否真的建立，最后查日志确认具体报错原因，不要只凭单一现象下结论。
