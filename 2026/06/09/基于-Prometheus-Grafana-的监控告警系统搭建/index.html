基于 Prometheus + Grafana 的监控告警系统搭建

发表于2026-06-09|更新于2026-06-09
|浏览量:

前言

在 DevOps 运维体系中，监控是保障系统稳定运行的核心环节。本文记录了在三台 CentOS 7 虚拟机上，使用 Docker Compose 部署 Prometheus + Grafana + Alertmanager 监控告警平台的完整过程，实现对主机与容器资源的实时监控。

环境规划

节点 IP 角色
monitor 192.168.2.107 Prometheus + Grafana + Alertmanager
node1 192.168.2.110 被监控节点（Node Exporter + cAdvisor）
node2 192.168.2.111 被监控节点（Node Exporter + cAdvisor）
  • 操作系统：CentOS 7.9
  • 容器运行时：Docker + Docker Compose

系统架构

整体架构分为四层：

  • 采集层：Node Exporter 采集主机指标（CPU、内存、磁盘、网络），cAdvisor 采集容器指标
  • 存储层：Prometheus 通过 pull 方式定时抓取各节点 /metrics 接口，存入本地时序数据库，保留 15 天
  • 告警层：Prometheus 评估告警规则，触发后推送至 Alertmanager，由 Alertmanager 负责路由、去重、静默和邮件通知
  • 展示层：Grafana 通过 PromQL 查询 Prometheus 数据，渲染可视化仪表盘

Dashboards → New → Import，输入 ID 1860（Node Exporter Full），选择 Prometheus 数据源后点 Import，即可看到完整监控面板。

最终效果

访问 http://192.168.2.107:3000，在 Node Exporter Full 仪表盘中可实时查看三台节点的 CPU 使用率与负载、内存使用情况、磁盘 I/O 与剩余空间、网络收发流量。

Grafana 仪表盘 http://192.168.2.107:3000
Prometheus 查询 http://192.168.2.107:9090
Alertmanager http://192.168.2.107:9093

总结

本项目从零搭建了完整的监控告警闭环：指标采集 → 时序存储 → 规则评估 → 告警通知 → 可视化展示。实际操作中遇到了 YAML 语法错误、防火墙拦截、镜像源失效等典型运维问题，逐一排查解决，加深了对 Prometheus 配置结构和 Docker 网络的理解。
